## Слайд 3 · Решение (2:00 — 2:45, 45 сек)

> Ключевая идея решения — **параллельный опрос шести независимых источников и потоковая выдача результатов** пользователю по мере готовности каждого из них.
>
> На схеме показан полный пайплайн: пользователь вводит запрос, backend через `asyncio.gather` стартует шесть парсеров одновременно, и каждый оффер сразу же отдаётся клиенту через Server-Sent Events. Клиент не ждёт самого медленного парсера — первая карточка появляется через **1,4 секунды**.
>
> Параллельно работает **AI-слой** на Gemini 2.5 Flash — отвечает на вопрос «дорого это или дёшево», опираясь на историю цен.

## Слайд 5 · Архитектура (3:15 — 4:15, 60 сек)

> Технически система построена на пяти слоях.
>
> **Клиент** — Next.js 14 на TypeScript со статическим экспортом. Использует EventSource для приёма SSE-потока и Zustand для инкрементального обновления карточек.
>
> **Edge** — nginx на VPS с поддержкой HTTP/2 и ALPN. Критично — отключено `proxy_buffering`, иначе nginx копит ответ и ломает стриминг.
>
> **Backend** — FastAPI на Python 3.11, асинхронный SQLAlchemy 2.0. Ядро — `asyncio.gather` по шести парсерам с `yield` каждого оффера в SSE-поток.
>
> **Парсеры и источники** — шесть отдельных async-модулей с единым контрактом `search(query) → list[Offer]`. *[Очереди задач — Celery поверх Redis, база — PostgreSQL 16.]*
>
> Всё развёрнуто как шесть Docker-контейнеров на одном VPS-сервере.

## Слайд 8 · «Найти дешевле» — реверс WebSocket Алисы (5:25 — 6:25, 60 сек)

> Это уникальная функция системы, **аналогов которой на рынке нет**.
>
> Идея: товар прогоняется через **закрытый WebSocket-канал бэкенда Яндекс.Алисы** — тот самый, через который голосовой ассистент находит «то же самое дешевле». Публичного API у этого канала нет, поэтому в рамках работы выполнен **исследовательский reverse-engineering протокола**.
>
> Алгоритм — три шага. Открываем WebSocket-сессию с UUID-эмуляцией клиента Яндекс. Парсим бинарный payload, извлекая типизированные сущности — `EAliceOfferCard`, `EAliceOffer`, `EAliceOfferList`. Мерджим найденные офферы в общую ленту по артикулу или EAN.
>
> Справа на слайде — реальный скриншот: запрос **iPhone 17 Pro**, лучшая найденная цена **119 090 рублей**, что на **16 113 рублей дешевле**, чем на Ozon — то есть минус двенадцать процентов на конкретном кейсе. Фича работает в продакшене и даёт измеримый экономический эффект.

## Слайд 13 · Инженерные вызовы (8:35 — 9:30, 55 сек)

> Четыре основных инженерных вызова, ради которых работа и существовала.
>
> **Первое — гетерогенные антибот-защиты шести маркетплейсов.** Один универсальный прокси не работает: для каждого источника свой стек. Onliner — открытый JSON API, Яндекс.Маркет — headless-браузер с прокруткой страницы для подгрузки SKU, Регард, 1click и BigGeek — HTTP-клиент с ротацией User-Agent и timing-jitter, World Devices — обход кэша миниатюр. В итоге — **шесть парсеров стабильно на проде, около девяноста пяти процентов успеха** на типовых запросах.
>
> **Второе — потоковый pipeline.** Параллельный опрос через `asyncio.gather` плюс потоковая отдача через `EventSourceResponse`. Главная грабля — nginx по умолчанию буферизует ответ и ломает SSE. Конфигурация `proxy_buffering off`, `gzip off` на стримовых роутах, HTTP/2 ALPN. **Первая карточка через 1,4 секунды.**
>
> **Третье — реверс бинарного WebSocket-протокола Яндекс.Алисы**, о чём я уже говорил.
>
> **Четвёртое — intent-aware фильтрация.** Был баг: поиск «чехол iPhone 16 Pro» возвращал четыре результата вместо двадцати пяти, потому что парсеры безусловно отсеивали аксессуары. Решение — детерминистический intent-детектор по семидесяти пяти regex-паттернам до обращения к LLM. **Результат на проде: четыре стало двадцать пять, Яндекс.Маркет с нуля поднялся до двадцати.**

## 🛡️ Заготовки ответов на вопросы

### «Почему SSE, а не WebSocket?»

> Серверу нужна только односторонняя передача — поток событий «нашёл, нашёл, нашёл». SSE проще, держит автореконнект из коробки, прозрачно проксируется через nginx и не требует дополнительной библиотеки на фронте. WebSocket был бы избыточен.

### «Реверс WebSocket Алисы — это законно?»

> Это исследовательский reverse-engineering в рамках академической работы. Условия использования не нарушены — мы работаем с трафиком собственного браузерного клиента, который любой пользователь Яндекс может наблюдать у себя в DevTools.

### «А если Яндекс изменит протокол?»

> Признаваемый риск. Функция «Найти дешевле» вынесена в отдельный воркер, изолирована от ядра. Если протокол меняется — отключается только эта фича, основной метапоиск по шести источникам продолжает работать. Предусмотрен мониторинг успешности WS-сессии.

### «Почему Gemini, а не GPT-4 или Claude?»

> Gemini 2.5 Flash через OpenRouter дешевле GPT-4o Mini в полтора раза и Claude Haiku в два с половиной раза, при сопоставимом качестве для tool-use и нормализации текста. Кроме того, OpenRouter решает проблему геоблокировки прямого Gemini API из России. Через OpenRouter можно горячо переключиться на любую другую модель без изменения кода.

### «А что с Wildberries и Ozon — почему их нет?»

> Wildberries был временно отключён из-за нестабильности полей наличия и попадания в выдачу восстановленных товаров вместе с новыми. Ozon — изначально не подключали из-за агрессивной антибот-защиты, требующей отдельной инфраструктуры. Архитектура такова, что новый источник подключается как отдельный модуль — ядро системы менять не нужно.

### «Что такое intent-aware фильтр? Это вы добавили недавно?»

> Это последняя инженерная итерация перед защитой. Раньше система не различала запрос «iPhone» от запроса «чехол на iPhone» — для парсеров оба запроса означали «отсей всё, что выглядит как аксессуар». Я добавил детерминистический детектор намерения по семидесяти пяти regex-паттернам, который срабатывает **до** обращения к LLM. Это даёт стопроцентно воспроизводимое поведение и фикс реального продакшен-бага.

### «Какая средняя загрузка вашего VPS?»

> На пиках до пятидесяти процентов CPU при пяти-десяти одновременных поисках. Бутылочное горлышко — не процессор, а сами маркетплейсы: ответы парсеров приходят за 0,9–1,8 секунды. По памяти — занимаем около 1,5 ГБ из 4 ГБ доступных.

### «Как тестировали 97% точности AI-коррекции?»

> Сто типовых пользовательских запросов с намеренными опечатками, сленгом, транслитерацией — «айфон 16 пра», «сяоми 15», «плойка 5». Результат вручную сверялся с эталонной канонической формой.
