Обзор зарубежных агрегаторов цен. Google Shopping — сервис сравнения цен, интегрированный в поисковую систему Google и доступный в более чем 100 странах мира. Основные преимущества сервиса: колоссальный охват продавцов, глубокая интеграция с поисковой системой, развитая инфраструктура индексирования товарных предложений. Вместе с тем Google Shopping имеет ряд существенных ограничений: крайне слабое покрытие российского рынка (отсутствие данных с Wildberries, Ozon, Яндекс Маркет), полное отсутствие белорусских магазинов, отсутствие функции отслеживания истории цен и AI-ассистента для анализа предложений. Бизнес-модель сервиса основана на платных листингах (Product Listing Ads), что создаёт конфликт интересов: наиболее заметные позиции занимают рекламодатели, а не объективно лучшие предложения [5].

Idealo.de — ведущий европейский агрегатор цен с охватом более 50 тысяч магазинов в Германии, Великобритании, Франции и других европейских странах. Сервис обладает развитой функциональностью истории цен, ценовыми оповещениями и детальными фильтрами. Основной недостаток — ограниченность европейским рынком и полное отсутствие покрытия рынков России и Беларуси. PriceRunner — скандинавский агрегатор, специализирующийся на рынках Северной Европы. CamelCamelCamel — специализированный трекер цен, работающий исключительно с платформой Amazon. Ни один из перечисленных зарубежных агрегаторов не покрывает рынки России и Беларуси, что подтверждает наличие незанятой ниши для разрабатываемого решения.

Обзор российских и белорусских сервисов. E-katalog.ru — наиболее известный российский агрегатор цен, функционирующий с 2006 года. Сервис предоставляет функциональность сравнения технических характеристик товаров, истории цен, рейтинги и отзывы. Покрытие включает крупные российские интернет-магазины и маркетплейсы. Яндекс Маркет занимает двойственную позицию: являясь одновременно маркетплейсом и агрегатором предложений сторонних магазинов, он создаёт потенциальный конфликт интересов. Кроме того, ни один из перечисленных сервисов не предоставляет возможности интерактивного диалога с AI-ассистентом для формулирования сложных поисковых запросов и получения аналитических рекомендаций.

Клиентские технологии. Для разработки клиентской части приложения выбран фреймворк Next.js 14 с App Router — современный React-фреймворк с поддержкой серверных компонентов, статического экспорта и оптимизированной маршрутизации. Режим статического экспорта (output: export) позволяет генерировать полностью статическое приложение, обслуживаемое через nginx без необходимости запуска Node.js-сервера в продакшене. Это существенно снижает требования к ресурсам сервера и упрощает развёртывание [9].

Ключевым технологическим прорывом для данной работы является механизм tool use (инструментальное взаимодействие, также известный как function calling). Данный механизм позволяет языковой модели не просто генерировать текст, но и формулировать структурированные запросы к внешним инструментам: базам данных, поисковым системам, API сторонних сервисов. Модель самостоятельно принимает решение о необходимости вызова инструмента, определяет параметры вызова, обрабатывает полученные результаты и формулирует ответ с учётом полученных данных.

Для данного проекта выбрана модель Gemini 2.5 Flash от компании Google, доступная через прокси-сервис OpenRouter. Основания выбора: высокое качество работы с русским языком (понимание контекста, грамматическая корректность, стилистическая адекватность), развитый механизм tool use с поддержкой множественных инструментов и итеративных вызовов, минимальная стоимость (0,10 доллара за миллион входных токенов), а также наличие потокового API для отображения ответов в реальном времени [13]. OpenRouter выступает прокси-сервисом, предоставляющим OpenAI-совместимый API и решающим проблему геоблокировки Gemini API в России. По сравнению с Claude Haiku ($0,25/M) и GPT-4o Mini ($0,15/M), Gemini 2.5 Flash обеспечивает сопоставимое качество при минимальной стоимости, что критично для проекта с ограниченным бюджетом.

Технологии контейнеризации и развёртывания. Docker обеспечивает воспроизводимость среды выполнения за счёт упаковки приложения вместе со всеми его зависимостями в изолированный контейнер. Это устраняет проблему works on my machine и гарантирует идентичное поведение приложения в средах разработки, тестирования и производства. Docker Compose расширяет возможности Docker, позволяя декларативно определять многоконтейнерные приложения в файле docker-compose.yml с указанием зависимостей, сетей, томов и переменных окружения [15].

Nginx выступает в роли высокопроизводительного обратного прокси-сервера и веб-сервера для обслуживания статических файлов. В контексте данного проекта nginx решает несколько задач: обслуживание статических файлов фронтенда, проксирование API-запросов к бэкенду, обеспечение корректной передачи SSE-потоков (отключение буферизации), реализация rate limiting для защиты от злоупотреблений, добавление заголовков безопасности [16]. Комбинация Docker Compose, nginx и VPS обеспечивает промышленный уровень развёртывания при минимальных затратах на инфраструктуру.

Обоснован выбор технологического стека: FastAPI (нативная асинхронность, SSE, Pydantic), Next.js 14 (App Router, статический экспорт), Gemini 2.5 Flash через OpenRouter (tool use, русский язык, стоимость), SSE (однонаправленная передача, автопереподключение), PostgreSQL 16 [20] (JSONB, pg_trgm, asyncpg), Redis 7 [21] (кэширование), Docker Compose (контейнеризация), nginx (обратный прокси). Выбранный стек является оптимальным для решения поставленных задач с учётом ресурсных ограничений проекта.

– Развёртываемость — полностью контейнеризованное развёртывание через Docker Compose на VPS с минимальными требованиями (2 ядра, 4 ГБ RAM).

Общая архитектура системы. Система Smart Price построена на основе многоуровневой клиент-серверной архитектуры с чётким разделением ответственности между компонентами. Архитектура включает четыре основных уровня: уровень представления (frontend), уровень бизнес-логики (backend API), уровень данных (PostgreSQL, Redis) и уровень внешних интеграций (маркетплейсы, AI API). Каждый уровень изолирован в собственном Docker-контейнере, что обеспечивает независимость развёртывания и масштабирования.

Уровень представления реализован как статическое одностраничное приложение (SPA) на Next.js 14 с App Router. Приложение собирается в статические HTML/CSS/JS файлы командой next export и обслуживается веб-сервером nginx без необходимости запуска Node.js в продакшене. Это существенно снижает потребление ресурсов сервера и упрощает развёртывание.

Уровень внешних интеграций обеспечивает взаимодействие с маркетплейсами (Onliner API, Яндекс Маркет, Wildberries API, Регард, World Devices) посредством специализированных парсеров, а также с AI-моделями через OpenRouter — прокси-сервис, предоставляющий единый OpenAI-совместимый API для доступа к моделям различных провайдеров (Google Gemini, Anthropic Claude, OpenAI GPT). Использование OpenRouter решает проблему геоблокировки: прямой доступ к Gemini API заблокирован из региона размещения VPS (Россия), тогда как OpenRouter маршрутизирует запросы через свою инфраструктуру.

Контейнеризация и Docker Compose. Инфраструктура развёртывания системы Smart Price организована на основе Docker Compose и включает четыре сервиса: backend (FastAPI-приложение на Uvicorn), postgres (PostgreSQL 16 Alpine), redis (Redis 7 Alpine с AOF-персистентностью и ограничением памяти 128 МБ) и nginx (обратный прокси и веб-сервер для статических файлов). Все сервисы объединены в единую внутреннюю Docker-сеть (bridge driver), а внешний доступ осуществляется исключительно через nginx (порты 80 и 443).

Сервис backend собирается из многоэтапного Dockerfile.prod (multi-stage build): первый этап устанавливает зависимости Python в виртуальное окружение, второй — копирует только необходимые файлы и виртуальное окружение, минимизируя размер итогового образа. Переменные окружения (DATABASE_URL, REDIS_URL, OPENROUTER_API_KEY, AI_MODEL) передаются через секцию environment в docker-compose.yml. Сервис postgres использует health check для обеспечения готовности базы данных перед запуском backend (depends_on с condition: service_healthy).

Статические файлы фронтенда монтируются в контейнер nginx через volume: ./frontend/out:/usr/share/nginx/html:ro. Конфигурация nginx обеспечивает: обслуживание статических файлов, проксирование API-запросов к бэкенду, отключение буферизации для SSE-потоков (proxy_buffering off), rate limiting для защиты от злоупотреблений, добавление заголовков безопасности (X-Frame-Options, X-Content-Type-Options, Content-Security-Policy).

Структура приложения Next.js. Фронтенд приложения реализован на Next.js 14 с использованием App Router — файловой системы маршрутизации, в которой структура директорий определяет URL-пути. Корневой layout.tsx определяет общую обёртку приложения: подключение шрифтов, метатеги, боковую навигацию. Каждая страница реализована как отдельная директория в app/ со своими page.tsx и layout.tsx:

Страница детальной информации о товаре. Страница детальной информации о товаре реализована как клиентский маршрут Next.js (/product) с динамической загрузкой данных по идентификатору товара, передаваемому в параметре URL key. Маршрут получает данные через эндпоинт GET /api/v1/onliner/product/{key} и параллельно запрашивает дополнительные внешние отзывы (/api/v1/external-reviews), что позволяет сократить общее время загрузки за счёт совмещения двух сетевых запросов.

Для обеспечения отзывчивости интерфейса реализована двухуровневая система состояний загрузки. Первый уровень — маршрутный Suspense fallback (файл product/loading.tsx), отображаемый Next.js моментально при переходе пользователя со страницы поиска на страницу товара. Это устраняет визуальное «зависание» интерфейса между моментом клика по карточке и началом монтирования компонента страницы. Второй уровень — внутренний полноэкранный лоадер компонента, активный во время выполнения запроса к API. Оба лоадера используют единый визуальный стиль (анимированный орбитальный индикатор «SP» с пульсирующей обводкой), что обеспечивает целостность пользовательского опыта и устраняет визуальные «дребезжания» при переключении между состояниями.

Выбор AI-модели и провайдера. В процессе разработки системы была выполнена миграция AI-модели с Claude Haiku (Anthropic) на Gemini 2.5 Flash (Google) через прокси-сервис OpenRouter. Основания миграции: стоимость Gemini 2.5 Flash ($0,10 за миллион входных токенов) в 2,5 раза ниже Claude Haiku ($0,25 за миллион), при сопоставимом качестве генерации для задач классификации и извлечения данных. При бюджете $10 система способна обработать приблизительно 25 000 поисковых запросов с AI-коррекцией и AI-фильтрацией.

OpenRouter выступает прокси-сервисом, предоставляющим единый OpenAI-совместимый API для доступа к моделям различных провайдеров. Это решает критическую техническую проблему: прямой доступ к Gemini API заблокирован из России (ошибка «User location is not supported for API use»), а VPS-сервер системы Smart Price расположен в российском дата-центре. OpenRouter маршрутизирует запросы через свою инфраструктуру, расположенную в США и Европе, обходя геоблокировку.

Назначение и архитектура подсистемы. Подсистема AI-анализа цен (модуль price_analyzer.py) реализует функцию интеллектуальной оценки выгодности предложения по запросу пользователя. В отличие от потокового поиска, который возвращает каталог найденных товаров, AI-анализ выполняет дополнительную статистическую и контекстную обработку: вычисляет ценовые показатели по выборке предложений, формирует развёрнутый аналитический комментарий на естественном языке и формулирует итоговый вердикт о выгодности предложения. Подсистема реализована в виде асинхронного сервиса класса PriceAnalyzer, доступного через эндпоинт GET /api/v1/analyze/stream в формате Server-Sent Events.

Архитектурно сервис представляет собой конвейер из шести последовательных этапов: валидация входных данных, параллельный сбор предложений с маркетплейсов, гибридная фильтрация результатов, статистическая обработка, обращение к большой языковой модели и формирование итогового ответа. Каждый этап завершается отправкой клиенту типизированного SSE-события, что позволяет фронтенду отображать прогресс обработки в реальном времени.

Конвейер обработки данных. Процесс анализа начинается с проверки региона (допустимые значения — BY и RU) и поискового запроса. После валидации сервис создаёт экземпляр ScrapingManager и параллельно запускает все включённые парсеры маркетплейсов, что обеспечивает минимизацию суммарного времени сбора данных. Сырые результаты объединяются в единый список и проходят трёхступенчатую фильтрацию.

Первая ступень — быстрый regex-фильтр (fast_filter), отсекающий заведомо нерелевантные позиции по совпадению ключевых токенов запроса. Вторая ступень — AI-фильтр релевантности (ai_filter_relevant), идентичный фильтру, применяемому при потоковом поиске; данная стадия использует языковую модель для семантической оценки соответствия найденных товаров запросу пользователя. Третья ступень — удаление ценовых выбросов (remove_price_outliers) и позиций с нулевой ценой. Если после фильтрации количество предложений оказывается меньше пяти, сервис возвращает ошибку «недостаточно данных для анализа» — данное пороговое значение выбрано экспериментально и обеспечивает статистическую достоверность последующих расчётов.

Статистический модуль. После фильтрации сервис вычисляет шесть ценовых показателей по итоговой выборке: минимум, максимум, медиана, среднее арифметическое, стандартное отклонение и количество предложений. Все расчёты выполняются средствами стандартной библиотеки Python (модуль statistics). Использование медианы наряду со средним позволяет получить устойчивую к выбросам оценку центральной тенденции, что критично в условиях гетерогенной ценовой выборки с разных маркетплейсов. Параллельно формируются три набора данных: лучшее предложение (best_offer — позиция с минимальной ценой), список из трёх альтернатив с ценой ниже медианы (cheaper) и список из трёх предложений с ценой выше медианы (pricier). Эти данные передаются клиенту в составе финального ответа и одновременно используются как контекст для языковой модели.

Интеграция с большой языковой моделью. Языковая модель Gemini 2.5 Flash вызывается через прокси-сервис OpenRouter с использованием унифицированного клиента OpenAI SDK. Системный промпт содержит набор строгих правил, направленных на предотвращение типичных проблем работы с LLM в задачах ценовой аналитики. Во-первых, модели запрещено выдумывать, округлять или интерпретировать численные значения цен — все числа поступают пользователю напрямую из статистического модуля. Во-вторых, в текстовом поле value_analysis запрещены любые цифры, проценты, упоминания валют и слова-заменители («примерно», «около»), что устраняет риск противоречия между числами в тексте LLM и реальными показателями выборки. В-третьих, ответ должен соответствовать строгой JSON-схеме, валидируемой моделью Pydantic LLMAnalyzePayload.

Протокол SSE-событий. Сервис эмитирует семь типов событий, последовательно отражающих ход обработки запроса: start — подтверждение приёма запроса с указанием параметров (запрос и регион); parsing — начало параллельного опроса маркетплейсов со списком активных источников; scraped — завершение сбора данных с указанием общего количества найденных предложений; stats — публикация рассчитанных ценовых показателей; analyzing — начало вызова языковой модели; result — финальное событие, содержащее полный объект AnalyzeResult (вердикт, оценка, лучшее предложение, альтернативы, аналитический комментарий) и метаданные источника ответа; error — отправляется при недопустимом регионе или недостатке предложений после фильтрации.

Процесс развёртывания. Развёртывание системы на VPS-сервере выполняется по следующему алгоритму. На этапе подготовки фронтенда выполняется сборка статических файлов: npm run build в директории frontend/ генерирует оптимизированный набор HTML/CSS/JS файлов в директории out/. Применяется минификация JavaScript и CSS, tree shaking неиспользуемых зависимостей, оптимизация изображений. Результирующий размер бандла составляет 280–350 КБ (gzip), что обеспечивает время загрузки менее 1 секунды на мобильном 4G-соединении.

Собранные статические файлы и исходный код бэкенда переносятся на VPS-сервер через scp или git pull. На VPS выполняется команда docker compose up --build -d, которая собирает образ бэкенда из Dockerfile.prod, поднимает все четыре контейнера и подключает их к внутренней сети. Первый запуск включает инициализацию базы данных PostgreSQL (создание таблиц через Alembic миграции). Последующие обновления выполняются через docker compose up --build -d с пересборкой только изменённых образов.

Мониторинг и логирование. Логирование реализовано через библиотеку structlog, обеспечивающую структурированные JSON-логи с автоматическим включением контекста (timestamp, уровень, модуль, дополнительные поля). Уровень логирования настраивается через переменную окружения LOG_LEVEL (INFO для продакшена, DEBUG для отладки). Логи контейнеров доступны через docker compose logs с фильтрацией по сервису.

Мониторинг состояния сервисов обеспечивается health check в Docker Compose: PostgreSQL проверяется через pg_isready, Redis — через redis-cli ping. При сбое health check Docker автоматически перезапускает контейнер (restart: unless-stopped). Для мониторинга доступности сайта используется внешний сервис UptimeRobot, отправляющий HTTP-запросы к nginx каждые 5 минут и уведомляющий о недоступности через Telegram.

Назначение подсистемы. Функция «Найти дешевле» реализует сценарий повторного поиска для конкретной товарной позиции: пользователь указывает прямую ссылку на карточку товара маркетплейса Яндекс Маркет, после чего система в автоматическом режиме подбирает у сторонних продавцов предложения идентичного товара по меньшей цене. Функция дополняет основной сценарий потокового поиска и ориентирована на случаи, когда пользователь уже выбрал конкретную модель и нуждается в проверке, не является ли текущее предложение избыточно дорогим. Подсистема реализована в виде асинхронной задачи Celery, выполняющей длительное взаимодействие с внешним источником данных без блокировки HTTP-запроса.

Источник данных и протокол взаимодействия. В качестве источника данных используется внутренний канал подбора ценовых предложений, применяемый голосовым помощником Яндекс. Доступ реализован через постоянное соединение по протоколу WebSocket с публичным шлюзом. Клиент, реализованный в модуле workers/alisa.py, выполняет последовательность шагов: установление защищённого соединения, прохождение процедуры инициализации сессии, отправка текстового запроса на поиск более дешёвых предложений для заданного идентификатора товара (SKU), приём и декодирование потока сообщений rich_uicard и нормальное закрытие соединения. Каждое сообщение передаётся в двоичном формате и содержит вложенное JSON-поле json_data с типизированной структурой EAliceOfferCard, EAliceOffer или EAliceOfferList.

Обнаружение отказов и завершение задачи. Поскольку удалённый сервис способен возвращать текстовый отказ («пока не умею искать такие товары», «не поддерживается» и иные формулировки), подсистема реализует детектор отказов на основе регулярного выражения. При совпадении ответа с любым из шаблонов задача завершается со статусом rejected в среднем за пятнадцать-двадцать секунд, что предотвращает бессмысленное ожидание до таймаута в десять минут. Список формулировок отказа сформирован эмпирически по результатам наблюдения за реальными ответами сервиса для нетипичных запросов (напитки, кондитерские изделия, товары ограниченного оборота). Статус задачи публикуется по мере обработки через отдельное SSE-соединение и принимает значения idle, searching, complete, rejected, что позволяет клиенту корректно отображать прогресс.

Пользовательский интерфейс. Подсистема снабжена отдельной страницей /cheaper с набором состояний, визуально согласованных с графическим стилем проекта. Для каждого состояния задействована собственная векторная анимированная иллюстрация: копилка со скатывающимися монетами — для исходного ожидания ввода; катящаяся корзина — для активной фазы поиска; улыбающаяся монета — для успешного завершения с указанием найденной экономии; грустная панда — для отказа сервиса. Вспомогательная плашка в верхней части страницы информирует об ожидаемой продолжительности операции («примерно десять минут»), а нижняя плашка отображает реальное прошедшее время и сообщение о возможности закрыть страницу без потери результата. Таким образом, пользовательский опыт приведён в соответствие с особенностями длительного асинхронного сценария.

Выводы по третьей главе. В третьей главе представлена реализация всех ключевых компонентов системы метапоиска Smart Price. Реализованы шесть парсеров маркетплейсов с различными стратегиями извлечения данных: JSON API (Onliner, Wildberries), HTTP + regex (Яндекс Маркет), HTTP + HTML-парсинг (Регард, World Devices). Реализована двухуровневая AI-коррекция запросов (Gemini + Яндекс Спеллер), обеспечивающая распознавание сленга и исправление опечаток. Реализована функция «Найти дешевле», позволяющая по ссылке на товар Яндекс Маркета подобрать более дешёвые предложения у сторонних продавцов через асинхронную Celery-задачу с извлечением данных из вложенной структуры EAliceOfferList и детектированием отказов по регулярному выражению.

Тестовая среда. Тестирование проводилось в двух средах: локальной среде разработки (Windows 11, Node.js 20, Python 3.12) и продакшен-среде на VPS-сервере (Ubuntu 22.04, Docker Compose, 2 ядра, 4 ГБ RAM, IP 5.42.123.75). Продакшен-тестирование выполнялось через реальный домен с HTTPS, что обеспечивало идентичность условий реальной эксплуатации. Тестовые запросы включали реальные товары из категорий электроники: смартфоны (Samsung Galaxy S25 Ultra, iPhone 17 Pro, Xiaomi 15 Pro), ноутбуки (Lenovo ThinkPad, HP Spectre), телевизоры (Tuvio 55), проекторные экраны (Cactus Moto Expert 100).

Анализ потребления ресурсов. Мониторинг потребления ресурсов в продакшен-среде (VPS с 2 ядрами и 4 ГБ RAM) показал следующие результаты. Суммарное потребление оперативной памяти всеми контейнерами Docker Compose составило 650–900 МБ в зависимости от нагрузки: backend (FastAPI) — 200–350 МБ, PostgreSQL — 120–180 МБ, Redis — 40–80 МБ (с ограничением 128 МБ), nginx — 15–30 МБ. Загрузка CPU при единичном поисковом запросе достигала 25–40 процентов (за счёт параллельных HTTP-запросов и парсинга), при 5 одновременных запросах — 70–85 процентов.

Для объективной оценки разработанной системы проведено сравнительное тестирование с тремя существующими решениями: E-katalog.ru, Яндекс Маркет и Onliner.by. Тестирование выполнено на одинаковом наборе из 5 поисковых запросов для каждого сервиса. Оценивались: полнота результатов (количество маркетплейсов), актуальность цен, время отклика и наличие AI-анализа.

VPS-сервер (2 ядра, 4 ГБ RAM, 60 ГБ SSD, безлимитный трафик) арендуется у российского провайдера по тарифу 600 рублей в месяц (~$6,5). Данная конфигурация обеспечивает стабильную работу всех четырёх Docker-контейнеров с запасом ресурсов: суммарное потребление RAM в режиме простоя — 650 МБ (16 процентов от доступных 4096 МБ), при пиковой нагрузке — 900 МБ (22 процента).
