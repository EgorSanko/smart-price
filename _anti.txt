Исследуемые маркетплейсы существенно различаются по ключевым параметрам: объёму товарооборота, географическому покрытию, доступности программных интерфейсов и уровню антибот-защиты. Эти различия определяют необходимость разработки индивидуальных стратегий извлечения данных для каждой площадки, что является одной из ключевых технических задач настоящей работы.
----
Технологии веб-скрапинга и методы обхода антибот-защиты. Веб-скрапинг — автоматизированное извлечение данных из веб-страниц с последующей их структуризацией для дальнейшего использования. Существуют три основных подхода к реализации веб-скрапинга, каждый из которых обладает собственным набором преимуществ и ограничений [7].
----
Первый подход — использование HTTP-клиентов (httpx, requests, curl-cffi). Данный подход обеспечивает высокую скорость выполнения запросов и минимальное потребление ресурсов (10–20 МБ оперативной памяти на соединение). Вместе с тем HTTP-клиенты не способны обрабатывать JavaScript-код, что ограничивает их применимость для сайтов с динамической загрузкой контента. Кроме того, стандартные HTTP-клиенты уязвимы перед TLS-фингерпринтингом — методом идентификации клиента по характеристикам TLS-рукопожатия, позволяющим серверу отличить запрос от реального браузера от запроса скрипта.
----
Второй подход — headless-браузеры (Playwright [19], Puppeteer, Selenium). Headless-браузеры обеспечивают полную эмуляцию поведения реального пользователя: выполнение JavaScript-кода, обработку AJAX-запросов, взаимодействие с динамическими элементами страницы. Основной недостаток — высокое потребление ресурсов: каждый экземпляр браузера занимает от 200 до 500 МБ оперативной памяти, что делает параллельный запуск нескольких экземпляров проблематичным на серверах с ограниченными ресурсами. Время выполнения запроса также значительно выше: 6–10 секунд по сравнению с 1–2 секундами для HTTP-клиентов.
----
Третий подход — антидетект-библиотеки (Scrapling, nodriver, undetected-chromedriver). Данные библиотеки эмулируют TLS-отпечаток реального браузера без накладных расходов полноценного headless-браузера. Потребление памяти составляет 20–40 МБ, скорость сопоставима с HTTP-клиентами, при этом обеспечивается защита от TLS-фингерпринтинга. Недостаток — невозможность обработки сложного JavaScript-кода.
----
Со стороны маркетплейсов применяются следующие механизмы антибот-защиты: CAPTCHA (включая reCAPTCHA и hCaptcha), TLS-фингерпринтинг (JA3/JA4 отпечатки), rate limiting (ограничение частоты запросов), JavaScript challenge (проверка возможности выполнения JS-кода в клиентском браузере), анализ поведенческих паттернов (движения мыши, скорость скроллинга, паузы между действиями). Стратегии противодействия включают ротацию User-Agent строк и прокси-серверов, рандомизацию временных задержек между запросами, управление сессиями и cookie, а также использование библиотек для эмуляции TLS-отпечатков.
----
В рамках данной работы для каждого маркетплейса выбран оптимальный подход с учётом его технических особенностей: для Onliner.by и Wildberries используется HTTP-клиент httpx (открытые/полуоткрытые API), для Яндекс Маркет — HTTP-клиент с regex-извлечением данных из HTML, для Ozon — каскадная стратегия с переключением между HTTP-клиентом, облачным headless-браузером и локальным Playwright.

Tailwind CSS [18] выбран для стилизации интерфейса благодаря утилитарному подходу, обеспечивающему высокую скорость разработки и минимальный размер итогового CSS-бандла (неиспользуемые стили автоматически удаляются при сборке). Zustand [23] используется для управления глобальным состоянием приложения: выбранный регион, результаты поиска, фильтры. TanStack React Query [24] обеспечивает кэширование серверных данных и автоматическое обновление при повторных запросах. Recharts [25] выбран для визуализации истории цен благодаря декларативному API и хорошей интеграции с React. Библиотека framer-motion обеспечивает плавные анимации переходов и появления элементов интерфейса.

WebSocket обеспечивает полнодуплексное двустороннее соединение между клиентом и сервером, позволяя обеим сторонам отправлять данные в произвольный момент времени. Данная технология оптимальна для чатов, онлайн-игр и других приложений, требующих двусторонней коммуникации. Основные недостатки: сложность реализации, проблемы с прокси-серверами и балансировщиками нагрузки, необходимость ручного управления переподключением при разрыве соединения.
----
SSE (Server-Sent Events) обеспечивает однонаправленную передачу данных от сервера к клиенту по стандартному протоколу HTTP. SSE оптимален для сценариев метапоиска по следующим причинам: однонаправленный характер передачи данных (сервер отправляет результаты клиенту), работа через стандартный HTTP без проблем с прокси-серверами, встроенный механизм автоматического переподключения при разрыве соединения, поддержка типизированных событий (разные типы данных в одном потоке), простота реализации как на сервере, так и на клиенте.
----
Nginx выступает в роли высокопроизводительного обратного прокси-сервера и веб-сервера для обслуживания статических файлов. В контексте данного проекта nginx решает несколько задач: обслуживание статических файлов фронтенда, проксирование API-запросов к бэкенду, обеспечение корректной передачи SSE-потоков (отключение буферизации), реализация rate limiting для защиты от злоупотреблений, добавление заголовков безопасности [16]. Комбинация Docker Compose, nginx и VPS обеспечивает промышленный уровень развёртывания при минимальных затратах на инфраструктуру.

Обоснован выбор технологического стека: FastAPI (нативная асинхронность, SSE, Pydantic), Next.js 14 (App Router, статический экспорт), Gemini 2.5 Flash через OpenRouter (tool use, русский язык, стоимость), SSE (однонаправленная передача, автопереподключение), PostgreSQL 16 [20] (JSONB, pg_trgm, asyncpg), Redis 7 [21] (кэширование), Docker Compose (контейнеризация), nginx (обратный прокси). Выбранный стек является оптимальным для решения поставленных задач с учётом ресурсных ограничений проекта.

Уровень бизнес-логики представлен FastAPI-приложением, запущенным под управлением ASGI-сервера Uvicorn. FastAPI обрабатывает REST API запросы, управляет SSE-потоками для стриминга результатов поиска и ответов AI-ассистента, координирует работу парсеров маркетплейсов и взаимодействует с AI-моделями через OpenRouter API. Асинхронная природа FastAPI позволяет эффективно обрабатывать множество одновременных запросов без блокировки.

Уровень данных включает PostgreSQL 16 для персистентного хранения структурированных данных (пользователи, товары, история цен, чат-сессии) и Redis 7 для кэширования результатов поиска и управления сессиями. Выбор PostgreSQL обусловлен поддержкой типа данных JSONB для хранения полуструктурированных данных о товарах, расширения pg_trgm для нечёткого текстового поиска и асинхронного драйвера asyncpg.

Уровень внешних интеграций обеспечивает взаимодействие с маркетплейсами (Onliner API, Яндекс Маркет, Wildberries API, Регард, World Devices) посредством специализированных парсеров, а также с AI-моделями через OpenRouter — прокси-сервис, предоставляющий единый OpenAI-совместимый API для доступа к моделям различных провайдеров (Google Gemini, Anthropic Claude, OpenAI GPT). Использование OpenRouter решает проблему геоблокировки: прямой доступ к Gemini API заблокирован из региона размещения VPS (Россия), тогда как OpenRouter маршрутизирует запросы через свою инфраструктуру.

Архитектура парсеров маркетплейсов. Система парсеров организована по паттерну «Стратегия» (Strategy Pattern): каждый маркетплейс реализует единый интерфейс поиска, а менеджер парсеров (ScrapingManager) координирует параллельное выполнение. Реестр парсеров хранится в виде словаря с метаданными: название маркетплейса, регион (RU/BY), цвет бренда для UI и флаг включения. На момент написания работы в системе зарегистрировано шесть парсеров:

– AI-ассистент (/api/v1/ai) — создание чат-сессии (POST /sessions), список сессий (GET /sessions), отправка сообщения с SSE-стримингом (POST /sessions/{id}/message).

Типографика построена на системном шрифтовом стеке (Inter, system-ui, sans-serif) с тремя уровнями размеров: заголовки (24–32px, font-weight 700), подзаголовки (16–18px, font-weight 600), основной текст (14px, font-weight 400). Компоненты интерфейса стилизованы с использованием утилитарных классов Tailwind CSS, что обеспечивает единообразие стиля и минимальный размер CSS-бандла за счёт автоматического удаления неиспользуемых стилей при сборке (tree shaking).

Контейнеризация и Docker Compose. Инфраструктура развёртывания системы Smart Price организована на основе Docker Compose и включает четыре сервиса: backend (FastAPI-приложение на Uvicorn), postgres (PostgreSQL 16 Alpine), redis (Redis 7 Alpine с AOF-персистентностью и ограничением памяти 128 МБ) и nginx (обратный прокси и веб-сервер для статических файлов). Все сервисы объединены в единую внутреннюю Docker-сеть (bridge driver), а внешний доступ осуществляется исключительно через nginx (порты 80 и 443).
----
Статические файлы фронтенда монтируются в контейнер nginx через volume: ./frontend/out:/usr/share/nginx/html:ro. Конфигурация nginx обеспечивает: обслуживание статических файлов, проксирование API-запросов к бэкенду, отключение буферизации для SSE-потоков (proxy_buffering off), rate limiting для защиты от злоупотреблений, добавление заголовков безопасности (X-Frame-Options, X-Content-Type-Options, Content-Security-Policy).
----
Типичные ошибки парсеров включают: таймаут HTTP-запроса (установлен в 30 секунд), антибот-блокировку (CAPTCHA, HTTP 403), изменение структуры HTML-страницы маркетплейса, сетевые ошибки. Для парсера Яндекс Маркет реализована каскадная стратегия: при ошибке HTTP-клиента автоматически активируется Playwright headless-браузер. Для остальных парсеров применяется однократная попытка с корректной обработкой ошибки.
----
Отказоустойчивость AI-компонентов. Все три AI-компонента системы (коррекция запросов, фильтрация результатов, AI-ассистент) спроектированы по принципу graceful degradation — при недоступности AI-модели система продолжает работу с пониженной функциональностью. AI-коррекция: при ошибке переключается на Яндекс Спеллер, при ошибке Спеллера — использует исходный запрос без коррекции. AI-фильтрация: при ошибке возвращает все результаты без фильтрации. AI-ассистент: при ошибке отображает сообщение с предложением повторить попытку.
----
Система кэширования реализована на базе Redis 7 с политикой вытеснения allkeys-lru (Least Recently Used) и ограничением памяти 128 МБ. Кэшируются три типа данных: результаты поиска товаров (TTL 10 минут — компромисс между актуальностью и снижением нагрузки на маркетплейсы), проксированные изображения товаров (TTL 1 час — изображения меняются редко), сессии пользователей (TTL 24 часа). При исчерпании памяти Redis автоматически удаляет наименее используемые ключи, что предотвращает ошибки переполнения.
----
Преимуществом данного подхода является высокая скорость и надёжность: время выполнения запроса составляет 300–800 миллисекунд, а структурированный формат ответа исключает ошибки парсинга HTML. Onliner.by применяет минимальную антибот-защиту, что позволяет использовать стандартный HTTP-клиент без дополнительных мер обхода.

Парсер Яндекс Маркет. Яндекс Маркет не предоставляет публичного API, поэтому парсер реализован с использованием двух стратегий: HTTP-клиент с regex-извлечением данных из HTML (основной путь) и Playwright headless-браузер (резервный путь при срабатывании антибот-защиты). Основная стратегия выполняет GET-запрос к URL поиска Яндекс Маркета с установленными заголовками User-Agent и Cookie для эмуляции обычного браузера. Из полученного HTML-ответа данные о товарах извлекаются с использованием регулярных выражений, нацеленных на JSON-LD-разметку и структурные элементы страницы.
----
При срабатывании антибот-защиты (CAPTCHA, JavaScript challenge) система автоматически переключается на резервную стратегию с использованием headless-браузера Playwright. Данный подход обеспечивает полную эмуляцию поведения пользователя, однако увеличивает время выполнения до 4–8 секунд и потребляет значительно больше ресурсов. Каскадный подход позволяет в 80–90 процентах случаев использовать быстрый HTTP-клиент и прибегать к Playwright только при необходимости.
----
Парсер Wildberries. Wildberries предоставляет внутренний JSON API поиска, обнаруженный методом реверс-инжиниринга сетевых запросов мобильного приложения. Парсер использует httpx для выполнения запросов к данному API с корректными заголовками, имитирующими мобильный клиент. Ответ представляет собой JSON с полями: наименование товара, цена (в копейках, требующая деления на 100), рейтинг, количество отзывов, идентификатор товара для формирования URL. Wildberries применяет rate limiting, поэтому парсер реализует рандомизированные задержки между запросами (1–3 секунды) для предотвращения блокировки.
----
World Devices — маркетплейс электроники с HTTP-доступным каталогом. Парсер аналогичен по архитектуре парсеру Регард: HTTP-запрос, HTML-парсинг, извлечение структурированных данных из карточек товаров. Особенностью данного маркетплейса является необходимость проксирования изображений через бэкенд системы Smart Price, поскольку World Devices блокирует запросы изображений с внешних доменов (защита от хотлинкинга).
----
Базовый агент и механизм tool use. AI-ассистент реализован по паттерну базового агента (BaseAgent), инкапсулирующему общую логику взаимодействия с языковой моделью через OpenRouter API. Базовый агент реализует цикл ReAct: получение сообщения пользователя, формирование запроса к AI-модели с контекстом диалога и описанием инструментов, обработка ответа модели, выполнение запрошенных инструментов, повторный запрос к модели с результатами инструментов, формирование финального ответа.
----
Каждое SSE-событие форматируется согласно спецификации: строка event: с типом события, строка data: с JSON-сериализованными данными и пустая строка-разделитель. Заголовки ответа включают Content-Type: text/event-stream, Cache-Control: no-cache и Connection: keep-alive. Конфигурация nginx включает директивы proxy_buffering off и proxy_cache off для предотвращения буферизации SSE-потока на уровне обратного прокси.
----
Для каждой страницы определены метаданные (metadata) через экспортируемый объект Metadata в layout.tsx: заголовок страницы формируется по шаблону «{название} | Smart Price», что обеспечивает корректное отображение в заголовке вкладки браузера и при индексации поисковыми системами. Фавикон реализован как SVG-файл (app/icon.svg) с фиолетовой буквой «S» на скруглённом фоне.

Управление состоянием. Глобальное состояние приложения управляется через Zustand — легковесную библиотеку управления состоянием для React, обеспечивающую минимальный размер бандла (менее 1 КБ) и простой API на основе хуков. Хранилище Zustand содержит: текущий регион (region), результаты поиска (searchResults), состояние загрузки (isSearching), выбранные фильтры (filters), данные текущего пользователя (user). Серверные данные (товары, история цен) кэшируются через TanStack React Query с автоматической инвалидацией при повторных запросах.

Выбор AI-модели и провайдера. В процессе разработки системы была выполнена миграция AI-модели с Claude Haiku (Anthropic) на Gemini 2.5 Flash (Google) через прокси-сервис OpenRouter. Основания миграции: стоимость Gemini 2.5 Flash ($0,10 за миллион входных токенов) в 2,5 раза ниже Claude Haiku ($0,25 за миллион), при сопоставимом качестве генерации для задач классификации и извлечения данных. При бюджете $10 система способна обработать приблизительно 25 000 поисковых запросов с AI-коррекцией и AI-фильтрацией.
----
OpenRouter выступает прокси-сервисом, предоставляющим единый OpenAI-совместимый API для доступа к моделям различных провайдеров. Это решает критическую техническую проблему: прямой доступ к Gemini API заблокирован из России (ошибка «User location is not supported for API use»), а VPS-сервер системы Smart Price расположен в российском дата-центре. OpenRouter маршрутизирует запросы через свою инфраструктуру, расположенную в США и Европе, обходя геоблокировку.
----
JWT-аутентификация. Система аутентификации реализована на основе JSON Web Tokens (JWT) с двухтокенной схемой: access token (срок действия 60 минут) для авторизации API-запросов и refresh token (срок действия 7 дней) для обновления access token без повторного ввода пароля. Пароли хранятся в базе данных в виде bcrypt-хешей с автоматическим salt, что исключает возможность восстановления пароля из хеша даже при компрометации базы данных.
----
Защита эндпоинтов реализована через зависимость FastAPI (Depends): декоратор get_current_user извлекает токен из заголовка Authorization: Bearer, верифицирует подпись, проверяет срок действия и возвращает объект пользователя. Эндпоинты, требующие аутентификации (AI-ассистент, профиль, история поиска), объявлены с данной зависимостью. Публичные эндпоинты (поиск, каталог) доступны без аутентификации.
----
Безопасность API. Для обеспечения безопасности API реализован ряд мер. CORS (Cross-Origin Resource Sharing) настроен с ограниченным списком разрешённых источников: домен продакшен-сервера и localhost для разработки. Заголовки безопасности добавляются через nginx: X-Frame-Options: DENY (защита от clickjacking), X-Content-Type-Options: nosniff (защита от MIME-sniffing), Strict-Transport-Security (принудительное использование HTTPS). Rate limiting реализован на уровне nginx: ограничение количества запросов к API до 30 в минуту с одного IP-адреса, а к эндпоинту авторизации — до 5 в минуту для защиты от brute-force атак.
----
Валидация входных данных реализована через Pydantic-модели: все параметры запросов типизированы и валидируются автоматически. SQL-инъекции предотвращены использованием SQLAlchemy ORM с параметризованными запросами. XSS-атаки предотвращены экранированием пользовательского ввода на уровне React (автоматическое экранирование в JSX) и Content-Security-Policy в заголовках nginx.

3.9 Реализация прокси-сервера изображений
----
Для корректного отображения изображений товаров с внешних маркетплейсов реализован прокси-сервер изображений (image_proxy.py). Необходимость прокси обусловлена тем, что некоторые маркетплейсы (World Devices, частично Wildberries) блокируют запросы изображений с внешних доменов через проверку заголовка Referer (защита от хотлинкинга). Прокси-сервер выполняет запрос к источнику изображения от имени бэкенда, добавляя корректные заголовки Referer и User-Agent, и передаёт полученное изображение клиенту.
----
Прокси реализован как FastAPI-эндпоинт GET /api/v1/image-proxy с параметром url (URL исходного изображения). Для безопасности реализована проверка списка разрешённых доменов: запросы принимаются только для известных маркетплейсов (wildberries.ru, onliner.by, world-devices.ru, regard.ru и др.). Изображения кэшируются в Redis с TTL 1 час, что снижает нагрузку на маркетплейсы при повторных просмотрах и ускоряет загрузку страниц.
----
Для повышения надёжности предусмотрено два независимых механизма отказоустойчивости. Первый — повторный запрос к модели при сетевой ошибке или некорректном JSON-ответе (максимум две попытки). Второй — детерминированный fallback (метод _fallback_payload), применяемый при полном отказе LLM-провайдера: вердикт и оценка вычисляются по формуле отклонения лучшей цены от медианы (score = max(0, 100 − deviation_pct)), а развёрнутый комментарий заменяется коротким автоматическим описанием. Источник итогового ответа («llm» или «fallback») указывается в метаданных финального события, что позволяет инструментам автоматического тестирования отличать эти два режима.
----
Назначение подсистемы. Функция «Найти дешевле» реализует сценарий повторного поиска для конкретной товарной позиции: пользователь указывает прямую ссылку на карточку товара маркетплейса Яндекс Маркет, после чего система в автоматическом режиме подбирает у сторонних продавцов предложения идентичного товара по меньшей цене. Функция дополняет основной сценарий потокового поиска и ориентирована на случаи, когда пользователь уже выбрал конкретную модель и нуждается в проверке, не является ли текущее предложение избыточно дорогим. Подсистема реализована в виде асинхронной задачи Celery, выполняющей длительное взаимодействие с внешним источником данных без блокировки HTTP-запроса.
----
Стратегия тестирования. Тестирование системы Smart Price проводилось по комплексной методологии, включающей четыре уровня: функциональное тестирование отдельных модулей, интеграционное тестирование взаимодействия компонентов, сквозное (end-to-end) тестирование через пользовательский интерфейс и нагрузочное тестирование производительности. Для автоматизированного сквозного тестирования использовался фреймворк Playwright, позволяющий эмулировать реальные пользовательские сценарии в headless-браузере Microsoft Edge.
----
Метрики времени отклика. Производительность системы оценена по ключевой метрике TTFR (Time To First Result) — времени от отправки поискового запроса до отображения первого товара. Измерения выполнены для серии запросов в продакшен-среде (smrt-price.ru) через автоматизированный фреймворк Playwright в браузере Microsoft Edge. Среднее значение TTFR составило 19,6 секунды, медианное значение — 18,8 секунды, 95-й перцентиль — 24 секунды.

Все выявленные дефекты были исправлены и верифицированы повторным автоматизированным обходом. Результаты подтверждают корректность функционирования всех основных пользовательских сценариев.

Обнаружено одно незначительное отличие: в Firefox анимация появления карточек (framer-motion) выполнялась с микро-задержкой (~50 мс), не влияющей на пользовательский опыт.
